## What Was Implemented

### 1. OpenAIConfig and AnthropicConfig Dataclasses

Defined minimal configuration classes for OpenAI and Anthropic providers (Phase 2/3 forward declaration):

**OpenAIConfig:**
- `api_key: str | None` with `metadata={"cli": False}` (excluded from CLI generation)
- `model: str = "gpt-4o-mini"` (sensible default)
- `base_url: str | None = None` (supports Azure/OpenAI-compatible gateways per Q18)
- `timeout_seconds: int = 60`
- `parameters: OpenAIParameters` with `temperature`, `top_p`, `max_tokens`

**AnthropicConfig:**
- `api_key: str | None` with `metadata={"cli": False}` (excluded from CLI generation)
- `model: str = "claude-3-5-sonnet-20241022"` (sensible default)
- `base_url: str | None = None` (trust boundary per Q18)
- `timeout_seconds: int = 60`
- `max_tokens: int = 4096` (required by Anthropic API, Q13)
- `parameters: AnthropicParameters` with `temperature`, `top_p`, `top_k`, `budget_tokens=1024` (Q12)

### 2. BackendConfig Discriminated Union

Updated `BackendConfig` to tagged-union shape:

```python
@dataclass(frozen=True)
class BackendConfig:
  provider: str = "ollama"
  ollama: OllamaConfig = field(default_factory=OllamaConfig)
  openai: OpenAIConfig | None = None
  anthropic: AnthropicConfig | None = None
```

**Validation:**
- `provider` validated against `("ollama", "openai", "anthropic")`
- `ollama` required when `provider == "ollama"` (Phase 1 guard)
- `openai` and `anthropic` may be `None` (forward-declared for Phase 2/3)

### 3. API Key CLI Exclusion (H1 Resolved)

All `api_key` fields annotated with `metadata={"cli": False}`:
- `OllamaConfig.api_key`
- `OpenAIConfig.api_key`
- `AnthropicConfig.api_key`

This ensures Clevis does NOT generate `--backend-ollama-api-key`, `--backend-openai-api-key`, or `--backend-anthropic-api-key` CLI arguments (security fix per Q6 amendment).

### 4. Backward Compatibility

**Old TOML files load unchanged:**
- Existing `~/.yoker.toml` files with single-Ollama shape continue to work
- Default `BackendConfig()` produces `provider="ollama"`, `ollama=OllamaConfig()`, `openai=None`, `anthropic=None`
- No migration script needed

## Design Decisions

1. **Discriminated dataclass over Union types**: Per Q2, using a discriminated dataclass with `provider` field allows Clevis to generate proper CLI arguments. Union types would break CLI generation.

2. **Provider whitelist**: `_ALLOWED_PROVIDERS = ("ollama", "openai", "anthropic")` validates provider choices early.

3. **Optional sub-configs**: `openai: OpenAIConfig | None = None` allows Phase 1 to ship with Ollama-only support while forward-declaring OpenAI/Anthropic for Phase 2/3.

4. **Validation guards**: Phase 1 only validates `ollama` is required when `provider == "ollama"`. Phase 2/3 will add equivalent guards for openai/anthropic.

5. **API key CLI exclusion**: Uses Clevis `metadata={"cli": False}` feature (released 2026-06-29) to address security finding H1.

6. **base_url trust boundary**: Per Q18 amendment, `base_url` fields exist but trust-boundary warning/confirmation is deferred to task 6.6 (startup path).
